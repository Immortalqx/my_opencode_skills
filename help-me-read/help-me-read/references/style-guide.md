# Style Guide for `help-me-read` Close-Read Notes

This file tells the writer how the note should *sound* and *feel*. Read it before writing the first paragraph of the note.

## 1. The Voice

The note is written in the user`s working language. The voice is **a patient tutor who has read the PDF carefully and is explaining it to a smart reader who has not**.

Concretely:

- Friendly, second-person sometimes ("you can now think about ..."), but mostly third-person and neutral.
- No academic stiffness, no marketing fluff.
- No "As we all know ..." or "It is well known that ...".
- No emoji decoration. The only block markers allowed are a single `> Warning ...` block per major topic for the most important caveat, and standard markdown callouts for source citations.
- No "Hope this helps" or "Feel free to ask" at the end. The note is a complete artifact, not a chat reply.

## 2. The Story Layer

The note opens with **one** running metaphor or story. The same metaphor is reused in every topic block. The metaphor is not a list, not a bullet summary, not a "5 things RL is like" list. It is **a single concrete scene with named characters**.

When the metaphor is chosen well:

- The reader can answer "What does X do in this story?" for every X in the document, without going back to the formal definition.
- The metaphor has a clear *before* state, a *problem* that breaks it, a *fix*, and a *result* --- exactly the structure of one topic block. This is why the metaphor carries the whole document.
- The metaphor suggests examples on its own. ("PPO is teaching a 5-year-old to ride a bike with soft training wheels" --- the reader can immediately imagine the child falling, the parent holding the handlebar, the "max 1 cm" rule, etc.)

The metaphor is **stated once, in section 1, in 3-5 sentences**. It is not re-stated in each section. Each section *uses* the metaphor to set up the motivation and the problem, then lifts the lid to show the formal math.

### What Makes a Bad Metaphor

- Too abstract ("RL is like learning" --- too vague to drive examples).
- Inconsistent mapping (a single object maps to two different concepts in different sections).
- One-shot jokes that do not scale (a quip that only works for the first topic).
- Stretched analogies that force the reader to ignore real differences.

### What Makes a Good Metaphor

- The characters have a clear goal, a clear failure mode, and a clear fix.
- Each character maps to one formal object (a policy, a reward, a value function, an advantage, a baseline, ...).
- The metaphor is *easy to draw*. If you cannot imagine illustrating it on a whiteboard, it is probably too abstract.

## 3. The Section Structure

For every topic block:

```
### N.M Topic Title (page references)

**Story beat.** 2-4 sentences that use the running story to set up *why this topic exists at all*.
The reader should feel a small "ah, that is the question" before any formal language appears.

**Problem.** The previous approach failed in X way. The previous approach is a named thing
(DQN, VPG, DDPG, ...), not a vague "old methods". Show the failure through the story.

**Solution.** Walk the reader through the new idea, *in the story first*, in 3-6 sentences.
Then present the formal definition, equation, or algorithm block, in a fenced code block.
Then give a 2-3 sentence reading of the formal block in plain words.

**Why this is enough.** The strongest argument for the design, plus the strongest argument against.
This is the place to mention caveats. One warning block is allowed here.

**Result.** Cite the PDF or a search result for the empirical evidence. Numbers are great;
"performs better in some setups" is not evidence.

**Example.** 1 short concrete scenario in the same story voice (e.g. a 7-DOF arm, a HalfCheetah,
a CartPole). If the PDF does not give one, search for one.
```

A topic block is **never** only a definition. A definition with no story beat is a sign that the topic was not understood. A topic block with no formal block is a sign that the formal part was skipped.

## 4. Term Glossing Rules

- Every key term is glossed the *first* time it appears in the note.
- Gloss format follows the user`s working language. When the note is in English, write the term in English. When the note is in Chinese, write the English term followed by a parenthetical Chinese gloss on first appearance.
- Do not gloss obvious words ("the", "is", "RL") or words the user obviously knows.
- Do not gloss a term twice. After the first appearance, use the same form consistently.
- For terms that are widely known in English but have a settled Chinese translation, prefer the **Chinese + English on first appearance**, then drop the English in subsequent mentions. The Chinese translation is generated by Codex at write time, not hard-coded in this skill.
- For terms that are still being argued about in Chinese, give both with a short note in parentheses.

## 5. Equation Rules

- Equations go in fenced code blocks, one block per equation. Use LaTeX-ish plain text or LaTeX depending on the renderer; the template shows the most portable form.
- Every equation is followed by:
  - one line that names the variables
  - one line that gives an intuition
  - one small example with concrete numbers
- A 10-line equation block with no intuition, no variable names, and no example is a failure mode. Break it up.

## 6. Figure Rules

There are two figure classes.

**Full-page slide screenshots** (for lecture slides):

- Render the entire page at 200 DPI or higher.
- Embed it as an image with a one-line caption.
- Use it when the slide *is* the visual. Do not crop a slide into a smaller figure unless the user asked for a region.

**Region screenshots** (for research papers):

- Crop a single figure or subfigure out of a page.
- Save the crop to `screenshots/selected/figure_<n>.png`.
- When the figure is a complex diagram (architecture, plot family, comparison table), call an image-understanding tool and store the explanation in `notes/figure_<n>_explanation.md`.
- Caption the embedded figure with the takeaway, not a description of the picture. Bad: "Figure 1: the model architecture." Good: "Figure 1: the actor outputs mu and sigma, and the critic estimates Q(s, a) from a reparameterized sample."

**Both classes share these rules:**

- The image path is relative to the note file. The note file and its asset folder sit next to each other.
- The asset folder is `final/<doc-slug>_<lang>_<version>_assets/`. Inside that folder, page screenshots live in `pages/`, cropped figures in `selected/`, and OCR/image-understanding outputs in `notes/`.
- If a figure is reused across notes, give it a stable name and a hash. Do not copy-paste the same crop into multiple asset folders.

## 7. Source Link Rules

- Every claim that came from outside the PDF has a working URL somewhere in the note. The default is a markdown link, written `[label](https://url)`.
- In-text links are good. A separate "References" section at the end is *also* required, so the reader can jump to all sources in one place.
- Search results are summarized into `search/results/<query-slug>.md` with the URL and a 1-3 sentence takeaway. The note may link to that local summary file *or* to the original URL, but at least one link to the original URL is required.
- Never invent a URL. If a search did not return a real source, say so in the note and drop the claim, or qualify it as "based on common usage in the literature".

## 8. The "Q&A" Block

A close-read note usually ends with a Q&A block: 9-15 user-style questions answered in the same story voice. The questions are the kind of thing the user might ask on a second or third read:

- "Why does this work, but only sometimes?"
- "What happens if we change one hyperparameter?"
- "How does this connect to the other topic?"
- "What did the author do that the previous author could not?"

Each Q&A is short (3-8 sentences). The answer is grounded in the running story, not just a definition. The Q&A is not a list of trivia; it is a list of "things the reader probably wondered about and was too embarrassed to ask".

## 9. What To Avoid

- "As we will see" followed by content that does not show the thing.
- Long enumerations of related work without a "why this matters here" beat.
- Quotations from the PDF longer than 2 sentences. Paraphrase or trim.
- Section openers that start with a definition. Always start with a story beat.
- "Hope this helps" / "Feel free to ask" at the end. The note is the artifact; the chat is over.
- Code blocks that are not runnable as written. If the code is illustrative, say so. If the code is runnable, keep it small and dependency-free.
- Reusing the exact same metaphor *and* the exact same sentence across multiple topic blocks. The story is the spine; the prose has to vary.

## 10. Length Budget

A close-read note for a 30-50 page lecture PDF should be 30-80 KB of markdown. A close-read note for a 10-15 page research paper should be 15-40 KB. If the note is much smaller, the running story is probably missing. If it is much larger, the topic blocks are probably not focused.

The Q&A block is 10-30% of the note. The Glossary is 3-8% of the note. The rest is the topic blocks.
